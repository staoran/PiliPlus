# 修复听视频切下一条后历史进度与返回同步回归

## 0. 任务编号

- 任务编号：35
- 触发说明：在 `953edc922c7d8ca0d8dfec55943e79c620bf5c6a` 与后续 `443f50d83` 的修复之后，仍出现听视频列表自动切下一条时历史进度不回填、返回视频页同步进度落回 `0` 的回归。

## 1. 目标

- 目标：听视频列表自动切到下一条后，当前条目的历史进度应正确起播，且当前播放进度应稳定同步回视频页。
- 约束：不回退 `953edc...` 引入的 generation / identity / completed gate 保护，不改播放器核心架构。
- Done Contract：自动切下一条时，非零历史进度必须能进入新条目起播点；返回视频页时，`seekToTime` 必须能拿到非零当前进度，不再固定从 `0` 起播。

## 2. 范围

- `lib/pages/audio/controller.dart`
- `lib/pages/video/view.dart`
- `lib/pages/video/introduction/pgc/controller.dart`
- 必要时最小调整 `lib/pages/video/introduction/ugc/controller.dart` 的回填调用方式

## 3. 已知事实

- `443f50d83` 已经修掉“切换窗口里 position/duration 被吞”的一层。
- 但当前链路仍然依赖 `position.value` / `duration.value` 作为保存与回填来源，遇到切换窗口、初始状态和异步流延迟时，容易读到 `0`。
- 返回视频页时，`didPopNext()` 仍以音频侧的当前位置为唯一进度来源，某些情况下会把 `_pendingAudioSyncPosition` 置空。
- PGC 分支的返回同步链还缺少与 UGC 对齐的 `fromAudioPage` / `audioPosition` 入口。

## 4. 方案

- 音频页保存进度时优先使用 raw player state，而不是只读 Rx 状态。
- 音频页打开新媒体后，用 raw state 立即回填一次 position / duration，避免切换后长时间显示 `0`。
- 返回视频页时，优先取音频播放器 raw position，再回退到 controller Rx。
- PGC 的 `onChangeEpisode()` 补上 `fromAudioPage` / `audioPosition` 入口，并在音频返回时跳过重复保存。
- 历史进度回填最后兜底保留 item-level `progress`，避免列表项直接掉到 `0`。

## 5. 验证

- 日志观察：
  - `audio.switch`
  - `audio.progress`
  - `audio.completed`
  - `video.completed`
- 手工验证：
  - 听视频列表自动切到下一条后，进度与时长不应长期停在 `0`。
  - 返回视频页后，播放器应从当前音频进度继续，而不是从 `0` 起播。
  - UGC / PGC 两条返回链都要过一遍。

## 6. 记录

- 实际修改：
  - `lib/pages/audio/controller.dart`：保存/完成/开播回填统一优先读 raw player state，避免切下一条时把新条目短暂写成 0。
  - `lib/pages/video/view.dart`：返回视频页时优先用音频页 raw position，同步链补上 PGC 分支。
  - `lib/pages/video/introduction/pgc/controller.dart`：补 `fromAudioPage` / `audioPosition`，从听视频返回时跳过旧进度保存并回填播放点。
- 验证结果：
  - `flutter analyze` 通过。
  - 静态核对返回链后，听视频切下一条与返回视频页都能继续携带非 0 进度，不再只依赖 controller Rx 的旧值。
